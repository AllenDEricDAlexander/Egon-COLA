# Web and Service Archetype Domain-First Package Design

Date: 2026-07-13

Status: Approved for design documentation. Implementation planning remains gated
until the user reviews this written spec.

## Goal

Change the generated Java package organization of these two archetypes from
technical-type-first to business-domain-first:

- `egon-cola-archetypes/egon-cola-archetype-web`
- `egon-cola-archetypes/egon-cola-archetype-service`

The generated Maven modules remain layer modules, but business code inside each
module must put the business domain before the technical responsibility.

For example:

```text
adapter/controller/user
application/command/course
domain/entities/user
infrastructure/repo/exam
```

becomes:

```text
adapter/user/controller
application/course/command
domain/user/entities
infrastructure/exam/repo
```

This work aligns the two multi-module archetypes with the domain-first convention
already used by `egon-cola-archetype-light`, while preserving the web and service
archetypes' existing module boundaries and runtime behavior.

## Confirmed Decisions

The user approved these decisions on 2026-07-13:

1. `model1` and `model2` are structural examples, not literal package names.
   - Web keeps the `user` and `teaching` business domains.
   - Service keeps the `course` and `exam` business domains.
2. Service remains service-only and does not gain a business Controller or other
   business HTTP surface.
3. Domain-first organization applies to production code, tests, package
   documentation, generated-project verification, and living generated-project
   documentation.
4. Truly cross-domain technical code may remain in a shared technical root.
   Domain-owned code must move into its owning business domain.
5. Cross-Project Facade clients are an explicit exception and retain their current
   technical-first integration boundary:
   - Web keeps `domain/client/evaluation` and
     `infrastructure/client/evaluation`.
   - Service keeps `domain/client/organization` and
     `infrastructure/client/organization`.
6. Facade interfaces live directly under the business domain, without an extra
   `api` package.
7. Converter packages use the spelling `converter`, not `convertor`.
8. Generated READMEs and the living architecture documents are updated. Historical
   specs and implementation plans remain unchanged.

## Design Authority and Precedence

Implementation resolves package-placement questions in this order:

1. The confirmed decisions in this spec.
2. The target package trees and shared-package rules in this spec.
3. The existing web and service architecture contracts for module ownership,
   dependency direction, runtime behavior, and cross-Project integration.
4. The existing archetype implementation where it does not conflict with the first
   three items.

This spec changes package organization only. It does not supersede existing
business behavior, module dependency, runtime-engineering, persistence, or
cross-Project Facade contracts.

## Scope

Implementation may change these living artifacts when required by package moves:

- Web and service files under
  `src/main/resources/archetype-resources`.
- Web and service files under `src/test/resources/projects/basic`.
- Web and service `META-INF/maven/archetype-metadata.xml` only if a file-set rule
  must change to generate the target paths correctly.
- The generated README owned by each archetype.
- `egon-cola-archetypes/egon-cola-archetype-web/multi-project-multi-module-architecture.md`.
- `egon-cola-archetypes/egon-cola-archetype-service/student-management-service-only-rpc-mq-architecture.md`.
- Repository CI only if it contains an assertion or command coupled to an old
  generated package path.

Implementation must not rewrite historical documents under:

```text
docs/superpowers/specs
docs/superpowers/plans
```

The only exception is this new spec and the later implementation plan created for
this work.

## Out of Scope

- Changing the seven generated Maven modules.
- Changing the generated internal Maven dependency graph.
- Renaming `user`, `teaching`, `course`, or `exam`.
- Adding business HTTP endpoints to the service archetype.
- Changing REST paths, GraphQL fields, Dubbo methods, Facade DTO fields, MQ
  semantics, error codes, or profile behavior.
- Changing cross-Project Facade coordinates or parameter names.
- Wiring the existing cross-Project clients into new Application use cases.
- Adding dependencies, frameworks, code generators, or architectural layers.
- Changing database schemas or data.
- Editing an existing Flyway migration or creating a new Flyway migration.
- Migrating projects that were generated by older archetype versions.
- Changing `egon-cola-archetype-light`.
- Starting a generated application, container, browser, or external service during
  validation.

## Package Organization Invariant

For business-owned code, every generated layer module uses this order:

```text
<layer>/<business-domain>/<technical-responsibility>
```

The following form is forbidden for local business domains:

```text
<layer>/<technical-responsibility>/<business-domain>
```

Therefore, these patterns must not remain:

```text
adapter/controller/user
adapter/mq/exam
application/command/course
application/result/teaching
domain/entities/user
domain/service/exam
infrastructure/repo/course
infrastructure/cache/teaching
facade/dto/user
```

The rule applies to package declarations, imports, source paths, test paths,
`package-info.java`, README trees, architecture examples, ArchUnit rules, and
`verify.groovy` path assertions.

The rule does not require artificial domain ownership for genuinely shared code.
It also does not override the approved external-client exception described below.

## Shared Package Rules

A type may remain in a shared technical package only when all of the following are
true:

1. It serves more than one local business domain or represents a Project-wide
   runtime concern.
2. Moving it into one business domain would create false ownership.
3. It does not contain a hidden collection of domain-specific implementations.

Mixed packages must be split. For example, a package containing User, Teaching,
and generic cache adapters keeps only the generic adapters at the shared root;
User and Teaching adapters move into their owning domains.

The implementation plan must identify the owner of every currently shared class
before moving it. It must not introduce `shared` packages merely to avoid making
that decision.

## External Client Exception

The user explicitly selected a technical-first exception for the existing
cross-Project clients. These paths remain valid:

```text
Web:
domain/client/evaluation
infrastructure/client/evaluation

Service:
domain/client/organization
infrastructure/client/organization
```

This exception is narrow:

- It applies only to the existing Evaluation and Organization Facade consumer
  boundaries.
- It does not permit local business packages such as `client/user`,
  `client/teaching`, `client/course`, or `client/exam`.
- External Facade imports remain confined to Infrastructure.
- Domain ports and projections remain consumer-owned and technology-neutral.
- Stub selection, timeout behavior, error mapping, Maven coordinates, and
  profile behavior remain unchanged.

Generated-contract checks must whitelist these four paths while rejecting other
business-domain-after-technical-package layouts.

## Source and Test Consistency

Every moved Java source must receive the matching package declaration and all
imports, fully qualified names, reflection strings, ArchUnit package patterns,
Spring scan assumptions, and test references must be updated.

Tests mirror the production domain-first structure where they test a specific
domain:

```text
src/test/java/.../adapter/user/controller
src/test/java/.../application/course
src/test/java/.../domain/exam
src/test/java/.../infrastructure/teaching/repo
```

Cross-domain architecture, profile, migration, bootstrap, and flow tests may
remain in shared test packages.

Every generated Java package contains a useful `package-info.java` consistent
with the existing archetype documentation style. Obsolete layer-first
`package-info.java` files and their empty directories are removed. The work does
not create empty symmetric domain packages when a domain has no implementation
for that technical responsibility.

## Generated Contract Verification

Both `verify.groovy` files must be updated as first-class generation contracts,
not treated as incidental path lists. They must verify:

- Required domain-first files and package documentation exist.
- Representative files exist in every affected layer and business domain.
- Old local layer-first paths are absent.
- The approved external-client exception remains present.
- Exact Maven module dependency sets remain unchanged.
- External Facade source-import boundaries remain unchanged.
- Generated READMEs describe the target package layout.
- No stale `convertor` package is introduced; the canonical spelling is
  `converter`.
- Service still contains no business Controller, Web, GraphQL, Filter, or VO
  package.

Where the verifier currently contains repeated literal path lists, the
implementation may use small data tables or helper functions already consistent
with the Groovy file's style. It must not weaken exact file and absence checks.

## Architecture and Design Pattern Assessment

The existing architecture already applies Ports and Adapters. The external
Facade clients also form an Anti-Corruption Layer. Both patterns remain unchanged.

No Strategy, Factory, Template Method, or other new runtime pattern is justified
for this work because package organization is the only changing concern. Adding
interfaces, factories, handlers, or inheritance would not improve the package
migration and would expand the requested scope.

The implementation should therefore be a direct, behavior-preserving package
migration backed by generated-contract tests.

## Runtime Data Flow and Error Handling

Runtime data flow does not change. Existing inbound Adapter, Application, Domain,
Infrastructure, and outbound Facade paths keep the same calls and dependency
directions; only their Java package names change.

Existing exception types, error codes, response wrappers, Facade failure mapping,
MQ failure semantics, and external-client failure categories remain unchanged.
Moving a type must not cause an exception to cross a module boundary that it did
not cross before.

## Compatibility

This is a generated-source contract change:

- New archetype invocations produce domain-first package names.
- Projects generated by older archetype versions are not modified automatically.
- Java package names in newly generated projects are intentionally incompatible
  with old source imports.
- Public route, RPC, DTO, database, and configuration contracts remain compatible.
- No compatibility aliases or duplicate old packages are generated because doing
  so would leave two competing package organizations in the sample.

## Validation

Implementation validation runs the smallest relevant checks first and does not
start either generated application.

### Static package audit

Search both archetypes for stale local layer-first paths, package declarations,
imports, README examples, and verifier assertions. The audit must distinguish the
approved external-client exception from forbidden local business paths.

### Archetype integration tests

```bash
bash ./mvnw -B -ntp \
  -pl egon-cola-archetypes/egon-cola-archetype-web \
  -am clean integration-test

bash ./mvnw -B -ntp \
  -pl egon-cola-archetypes/egon-cola-archetype-service \
  -am clean integration-test
```

These commands must generate the sample projects and execute their current
generation contracts without requiring an external runtime service.

### Generated project verification

For each generated project, run its Maven Wrapper verification using the local
repository setup produced by the archetype integration test when required by the
Facade fixtures. At minimum, compilation, tests, ArchUnit checks, Flyway checks,
and packaging checks already owned by the generated project must pass.

### Documentation and migration checks

- Confirm both generated READMEs and both living architecture documents use the
  domain-first examples.
- Confirm historical specs and plans were not rewritten.
- Confirm existing Flyway files are byte-for-byte unchanged and no migration was
  added.
- Confirm no browser, generated application, container, or external service was
  started.

## Acceptance Criteria

The work is complete only when all of the following are true:

1. Web local business packages use `user` or `teaching` before their technical
   responsibility in Facade, Domain, Application, Infrastructure, and Adapter.
2. Service local business packages use `course` or `exam` before their technical
   responsibility in the same modules.
3. Service contains no business Controller or other business Web surface.
4. Facade interfaces live directly under their business domain.
5. Converter packages use `converter` everywhere in these two archetypes.
6. Production and domain-specific test packages follow the same rule.
7. Shared packages contain only genuinely cross-domain concerns.
8. The four approved external-client paths retain their technical-first layout.
9. Old local layer-first paths are absent from templates and generated projects.
10. `package-info.java`, generated READMEs, living architecture documents,
    ArchUnit rules, and `verify.groovy` agree with the target structure.
11. Maven module dependencies and all existing behavior remain unchanged.
12. Both archetype integration tests and generated-project verification pass.
13. No existing Flyway migration is modified and no new migration is created.
14. Historical specs and plans remain unchanged.
15. Validation does not start either project or open a browser.

## Implementation Planning Boundary

After the user approves this written spec, a separate implementation plan will
divide the migration into reviewable tasks. Each implementation task will have a
non-overlapping scope, targeted validation, and one intentional commit in
accordance with the repository's agent instructions.

This spec does not authorize implementation before that plan is written and
reviewed through the next workflow stage.
