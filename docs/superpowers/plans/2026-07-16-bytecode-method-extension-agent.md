# Bytecode Method Extension Agent Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an explicit Agent engine for Method Extension that supports annotated concrete instance methods of every visibility, self-invocation, and non-Spring targets while keeping existing AOP behavior as the default.

**Architecture:** Extract the existing Handler/decision/response orchestration from `MethodExtensionAop` into one Spring-side execution service used by both engines. Agent-transformed methods call the JDK-only policy bridge at entry, and generated bytecode applies a typed `PROCEED`, `RETURN_NULL`, `RETURN_VALUE`, or `THROW` decision. The bytecode starter conditionally registers a Method Extension dispatcher only in Agent mode; the standalone Method Extension starter has no Agent/bridge dependency.

**Tech Stack:** Java 21 production, Java 21/25 forked verification, existing Method Extension Spring AOP component, existing bytecode platform, ASM 9.9.1, Spring Boot 3.5.16, Jackson from the application context, JUnit 5.

## Global Constraints

- Complete and merge Architecture Plugin, Agent/Executor, and Method Observation plans first.
- Follow `docs/superpowers/specs/2026-07-15-bytecode-enhancement-design.md` exactly.
- AOP remains default when `engine` is absent. `enabled=false` always disables both engines.
- Agent mode supports public/protected/package-private/private concrete instance methods, including final and synchronized methods.
- Agent mode excludes static methods, constructors, abstract methods, and native methods.
- Keep `MethodExtensionContext.target()` and `.method()` non-null.
- Reuse existing Handler resolution, decision validation, JSON conversion, and logging privacy; do not implement a second semantics path in bytecode modules.
- A method executes at most one engine and a Handler executes at most once per invocation.
- Allow decisions return the exact original business result, including the exact asynchronous object.
- Reactor types remain excluded.
- Method Extension starter must remain independently usable and must not depend on bytecode API/bridge/runtime/Agent/starter.
- Do not add Attach/retransform, database, Flyway, UI, browser, Docker, or application startup.
- Use an isolated worktree and one path-scoped commit per task.

## Pattern Boundary

Use an Adapter to feed Agent invocations into the existing Handler/response semantics and keep one domain execution service as the semantic seam. Do not add a second Strategy/Factory layer around Handlers or a generic policy chain; the repository already owns that variation point and direct orchestration preserves AOP parity.

## Stable Interfaces

Add these existing-component types:

```java
package top.egon.cola.component.methodextension.execution;

public record MethodExtensionExecutionResult(boolean proceed, Object rejectionValue) {
    public static MethodExtensionExecutionResult proceedInvocation() {
        return new MethodExtensionExecutionResult(true, null);
    }

    public static MethodExtensionExecutionResult reject(Object value) {
        return new MethodExtensionExecutionResult(false, value);
    }
}

public class MethodExtensionExecutionService {
    public MethodExtensionExecutionResult evaluate(Object target, Method invokedMethod, Object[] arguments)
            throws Throwable;
}
```

Properties use:

```java
public enum MethodExtensionEngine { AOP, AGENT, DISABLED }
public enum MethodExtensionNotReadyPolicy { PROCEED, REJECT, FAIL }
```

The Agent entry bridge is:

```java
InvocationDecision decision = EgonPolicyBridge.evaluateMethodExtension(
        target, DeclaringClass.class, methodId, boxedArguments);
```
