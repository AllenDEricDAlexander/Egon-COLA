# Bytecode Method Observation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add low-overhead observation for concrete instance/static methods and explicitly selected constructors outside Spring proxy boundaries, preserving original returns and Throwable identities.

**Architecture:** Observation matching is resolved by the Agent's single class enhancement planner from annotation/package/method/exclusion configuration. Generated bytecode calls the JDK-only observation bridge; the application runtime tracks timing and publishes bounded events, while the starter adapts them to Micrometer. Constructor observation begins only after the first successful `this(...)` or direct `super(...)` initialization call.

**Tech Stack:** Java 21 production, Java 21/25 verification, ASM 9.9.1, existing bytecode API/bridge/core/runtime/Agent/starter, Spring Boot 3.5.16, Micrometer 1.15.12, JUnit 5, JMH 1.37.

## Global Constraints

- Complete and merge the Architecture Plugin and Agent/Executor plans first.
- Follow `docs/superpowers/specs/2026-07-15-bytecode-enhancement-design.md` exactly.
- Support public/protected/package-private/private concrete instance methods, static concrete methods, final methods, synchronized methods, same-class calls, and non-Spring objects.
- Exclude abstract/native methods, `<clinit>`, synthetic/bridge/generated lambda bodies by default, and constructor code before successful initialization.
- Constructor observation is disabled unless annotated or `observe-constructors=true` is present in transform-time Agent configuration.
- Preserve the exact return value and exact Throwable instance.
- Never record arguments, return values, arbitrary object strings, credentials, identity data, headers, cookies, passwords, or tokens.
- Trace/request IDs may be event fields but never metric tags.
- No new Transformer registration: extend the existing composite planner/transformer.
- No Attach, retransform, byte dumps, database, Flyway, UI, browser, Docker, or application startup.
- Keep API/bridge JDK-only and runtime Spring-free.
- Use an isolated worktree and one path-scoped commit per task.

## Pattern Boundary

Use Strategy only for match policy and event sinks, plus the existing composite enhancement planner. Observation lifecycle remains direct bridge calls with one runtime service; do not add Observer buses, interceptor chains, or factory hierarchies because they would add allocation and reentrancy surface without another approved variation point.

## Stable Interfaces

```java
package top.egon.cola.component.bytecode.api.observation;

@java.lang.annotation.Target({java.lang.annotation.ElementType.METHOD,
        java.lang.annotation.ElementType.CONSTRUCTOR})
@java.lang.annotation.Retention(java.lang.annotation.RetentionPolicy.RUNTIME)
@java.lang.annotation.Documented
public @interface EgonObserved {
    String value() default "";
    String[] tags() default {};
    long slowThresholdMillis() default -1L;
}

public interface ObservationEventSink {
    void publish(ObservationEvent event);
}

public enum ObservationResult { SUCCESS, ERROR }
```

`ObservationEvent` contains `methodId`, class/method/descriptor metadata, layer, durationNanos, `ObservationResult`, exceptionGroup, traceId, virtualThread, and validated static tags. It contains no argument, return object, Throwable, or raw exception message.

Bridge lifecycle is fixed:

```java
ObservationToken token = EgonObservationBridge.enter(Owner.class, methodId);
try {
    Object value = originalBody();
    EgonObservationBridge.success(token);
    return value;
} catch (Throwable throwable) {
    EgonObservationBridge.error(token, throwable);
    throw throwable;
} finally {
    EgonObservationBridge.exit(token);
}
```
